What is LangChain?

LangChain is an open-source library designed to simplify building LLM-based applications. It provides modular building blocks for creating sophisticated LLM-based workflows.
Core Components
  • Model — unified interface for all LLM providers
  • Prompts — prompt engineering helpers
  • Retrievers — fetch docs from a vector store
  • Chains — biggest offering; connects everything
Use Cases
  • Chatbots & Text Summarizers
  • Multistep workflows
  • RAG applications
  • Basic-level agents

What is LangGraph?

LangGraph is an orchestration framework built on top of LangChain that enables the creation of stateful, multi-actor, and agentic applications using LLMs. Think of it as a flowchart engine where you define nodes (actions) and edges (control flow logic).
Core Principles
  • Graphs — models complex workflows as graphs instead of linear chains
  • State — robust state management via Pydantic/TypedDict
  • Control Flow — supports cycles, conditional branching, and looping
Key Capabilities
  • Human-in-the-loop (pause/resume)
  • Fault tolerance and check-pointing
  • Multi-agent orchestration
  • Event-driven architecture

Use Case: Automated Hiring Workflow

A real-world agentic workflow that shows where LangChain alone starts to struggle.

Full Workflow Steps
START
↓
Drafting JD (Reads docs → creates JD)
↓
HR Approval? (No → Ask for Changes / Loop)
↓ Yes
Posting JD (LinkedIn & Naukri APIs)
↓
Monitor Applications
↓
Revising JD? (Apps low? → Broaden to Full-Stack → Repost)
↓ Apps sufficient
Shortlisting (Parses resumes → Top 2)
↓
Scheduling (Checks Friday calendar → Invites)
↓
Interviewing (Sends reminders + questions)
↓
Offer Letter (HR Picks 1 → Draft → Approve → Send)
↓ Accepted
Onboarding (IT Access, Laptop, Meetings) → END
The Theory: Why Simple Chains Fail Here
  • Non-Linear Execution: Demands loops and conditional branching for retries and feedback.
  • Event-Driven Delays: Must pause execution to wait for external events and wake up reliably.
  • Human-in-the-Loop (HITL): Gracefully waits for human sign-off on critical business decisions.
  • Persistent Global State: Requires robust, shared memory to track data across disconnected steps.

Agents vs Workflows (Anthropic)

From Anthropic's "Building Effective Agents" (Dec 2024): Successful LLM implementations use simple, composable patterns over complex frameworks.
Workflows
  • LLMs + tools orchestrated via predefined code paths
  • Deterministic, sequential execution
  • Examples: chains, summarizers, RAG
VS
Agents
  • LLMs dynamically direct their own process & tool usage
  • Non-linear, event-driven
  • Examples: hiring agent, autonomous researcher

LangChain Challenges → LangGraph Solutions

1
Control Flow Complexity
LangChain Problem
  • Conditional branching is awkward
  • Loops must be written manually
  • Jumps between steps need "glue code"
  • Logic lives in raw Python while loops outside the chain
→
LangGraph Solution
  • Nodes, edges, and conditional edges built into the graph
  • No custom Python glue code needed
# LangGraph approach
graph.add_node("HiringRequest", hiring_request)
graph.add_node("CreateJD", create_jd)
graph.add_node("CheckApproval", check_approval)
graph.add_node("PostJD", post_jd)

graph.add_edge("HiringRequest", "CreateJD")
graph.add_edge("CreateJD", "CheckApproval")
graph.add_conditional_edges(
    "CheckApproval", approval_router,
    {"approved": "PostJD", "not_approved": "CreateJD"}  # Loop back!
)
graph.add_edge("PostJD", END)
2
Handling State
LangChain Problem
  • State managed manually — passed around as custom dict
  • Memory limited to conversation history
  • Complex workflows need manual state wiring
→
LangGraph Solution
  • Built-in stateful graph — typed state object (Pydantic / TypedDict)
  • Every node can read and edit the state
  • State covers the entire workflow

Example State Object

state = {
  "goal": "Hire a backend software engineer",
  "jd": "",
  "jd_approved": False,
  "jd_posted": False,
  "min_applicants": 5,
  "num_applications": 0,
  "shortlisted_candidates": {},
  "interview_questions": [],
  "offer_status": { "sent": False, "accepted": False, "renegotiated": False },
  "onboarding_status": { "completed": False, "start_date": None, "employee_id": None }
}
3
Event Driven Execution
LangChain Problem
  • Purely sequential: step 1 → 2 → 3 → end
  • Cannot pause mid-execution
  • Cannot wait for an external trigger
→
LangGraph Solution
  • Event-driven: can pause, wait for input, resume
  • Supports checkpointing: save state → store → resume later
  • Enables async human review without time constraints
4
Fault Tolerance
LangChain Problem
  • Chains are designed to be short
  • No built-in retry or recovery logic
  • Long-running workflows: one failure = restart from scratch
→
LangGraph Solution
  • Retry logic built-in at node level
  • Checkpointing via a persistence layer
  • On failure: resume from last checkpoint, not from start
5
Human in the Loop
LangChain Problem
  • Chain cannot pause mid-execution for human review
  • No mechanism to inject human input mid-workflow
→
LangGraph Solution
  • Workflow pauses at any node, collects human input, then continues
  • State saved to store (e.g. 24 hrs), human reviews/approves, execution resumes
LangGraph HIL Patterns:
  • Approve or Reject actions
  • Edit Graph State mid-execution
  • Review Tool Calls before execution
  • Persistent state (pause for minutes, hours, or days)
6
Nested Workflows (Subgraphs)
LangChain Problem
  • No native concept of composing graphs inside graphs
  • Multi-agent coordination is manual
→
LangGraph Solution
  • Subgraphs: a graph used as a node inside another graph
  • Enables multi-agent systems (e.g. self-driving car)
  • Supports reusability — subgraphs are reusable modules
7
Observability
LangChain Problem
  • Observability is partial — hard to monitor or debug complex flows
  • Glue code + while loops are opaque
→
LangGraph Solution
  • Native integration with LangSmith
  • Full timeline of node executions, state transitions, and tool calls
  • Monitor, debug, and understand the workflow at runtime
Observability = ability to monitor, debug, and understand what your workflow is doing at runtime.

Conclusion

What is LangGraph? An orchestration framework that enables you to build stateful, multi-step, event-driven workflows using LLMs. Think of it as a flowchart engine for LLMs — you define the steps (nodes), connections (edges), and transition logic. LangGraph handles state management, conditional branching, looping, pausing/resuming, and fault recovery.
Feature LangChain LangGraph
Control Flow Manual Python glue code Built-in nodes, edges, conditional edges
State Management Manual dict, limited memory Typed, stateful state object (Pydantic/TypedDict)
Execution Model Sequential only Event-driven, pausable, resumable
Fault Tolerance None — restart from scratch Retry logic + checkpointing + resume
Human-in-Loop Not supported mid-execution Native — pause, input, continue
Multi-agent Manual coordination Subgraphs + native multi-agent support
Observability Partial Full via LangSmith (timeline, traces)
Best For Simple, linear workflows Complex, non-linear, agentic workflows
Use LangChain When
  • Simple, linear workflows
  • Prompt chains
  • Text summarizers
  • Basic RAG / retrieval systems
Use LangGraph When
  • Conditional paths & loops
  • Human-in-the-loop steps
  • Multi-agent coordination
  • Async / event-driven execution
  • Long-running production workflows
Does LangGraph replace LangChain? No. LangGraph is built on top of LangChain. You still use LangChain components inside each node: ChatOpenAI, PromptTemplate, Retrievers, DocumentLoaders, Tools, etc.

LangGraph handles workflow orchestration. LangChain provides the building blocks for each step.
